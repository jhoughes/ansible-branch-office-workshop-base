# The Vault Password File

This workshop uses Ansible Vault to encrypt sensitive variables (a fake database connection string, an API key, a service account credential — section 3.2 covers this in detail).

To run any playbook that decrypts vaulted variables, Ansible needs to know the Vault password. The workshop's `ansible.cfg` is configured to read it from a file named `.vault_pass` in the workshop root.

## Setting up your Vault password file

**During the workshop:** Your instructor will hand you a card at the door with the Vault password. When you reach section 3.2, you'll copy the example file and paste your password in:

```bash
cp .vault_pass.example .vault_pass
nano .vault_pass     # or vim, or vi, or your editor of choice
# Replace the placeholder text with the password from your card.
# Save and exit.
```

**Important:** The `.vault_pass` file is in `.gitignore` and is NEVER committed to the repo. Only the `.vault_pass.example` placeholder is tracked in Git.

## Why a file instead of typing the password each time?

You *could* run every playbook with `--ask-vault-pass` and type the password each time. For a workshop where you'll run playbooks dozens of times in 4 hours, that gets old fast. The file approach is the standard production pattern for non-interactive runs (CI/CD systems, scheduled jobs, etc.) and it's what the workshop uses throughout.

## Don't lose this file or commit it

If you commit `.vault_pass` to a public repo by accident, every secret you ever encrypted with that password is now public. Treat it like an SSH private key: never commit, never email, never paste into Slack.

For this workshop the password is ephemeral — at the end of the day, the labs get torn down and the password becomes worthless. But the *habit* of treating Vault passwords as secrets is the lesson worth taking home.
